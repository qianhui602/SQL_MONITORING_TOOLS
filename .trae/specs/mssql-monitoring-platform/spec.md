# MS SQL 监控平台规范

## 为什么做这个
需要一个集中式平台来实时监控 MS SQL Server 2019 STD 的数据库健康状态、服务器内存使用情况，并自动捕获导致死锁的 SQL 语句，以便快速定位和解决性能问题。

## 变更内容

- 搭建一个基于 Web 的 SQL Server 监控平台
- 实现对目标数据库服务器（测试环境：10.239.253.3 / 生产环境：10.239.253.2）的连接与数据采集
- 监控 SQL Server 关键性能指标：CPU、内存、连接数、I/O
- 监控数据库服务器操作系统级别的内存使用状态
- 自动检测并记录数据库死锁事件及其对应的死锁 SQL 语句
- 支持历史数据存储与趋势展示
- 监控平台自身使用 PostgreSQL 作为后台数据库（10.239.254.14:5432）
- 提供告警通知机制（当内存使用过高或死锁发生时）
- 未来支持 Docker 容器化部署

## 影响范围

- 影响的能力：数据库运维监控、性能分析、故障排查
- 影响的代码/系统：
  - 数据采集层（SQL 查询脚本/Agent）
  - 后台数据存储层（PostgreSQL）
  - Web 展示层（Dashboard 面板）
  - 告警通知模块
  - Docker 部署配置（未来）

## 新增需求

### 需求 1：后台数据库（PostgreSQL）

监控平台自身使用 PostgreSQL 数据库存储所有采集数据和配置信息。

| 配置项 | 值 |
|-------|-----|
| 主机地址 | 10.239.254.14 |
| 端口 | 5432 |
| 数据库名 | sv_test_db |
| 用户名 | u_sv_mgt |
| 密码 | Sunvalleyco.com |
| 用途 | 存储性能指标、死锁事件、告警日志、系统配置 |

#### 场景 1.1：后台数据库连接成功
- **前提** PG 服务器可达，账号权限正确
- **操作** 监控平台启动时自动连接 PG
- **结果** 成功连接，自动建表（如不存在）

#### 场景 1.2：后台数据库连接失败
- **前提** PG 服务器不可达或认证失败
- **操作** 监控平台启动
- **结果** 记录错误日志，平台停止启动或进入降级模式

### 需求 2：数据库连接与数据采集（目标 SQL Server）

系统应能连接到目标 MS SQL Server 2019 STD，并定期采集以下指标。

**测试环境**：
- 地址：10.239.253.3
- 认证方式：SQL Server 身份验证
- 账号/密码：PowerBI / PowerBI

**生产环境**：
- 地址：10.239.253.2
- 认证方式：SQL Server 身份验证
- 账号/密码：PowerBI / PowerBI

#### 场景 2.1：连接成功
- **前提** 目标服务器可达，账号权限正确
- **操作** 系统启动采集任务
- **结果** 成功建立连接，开始采集指标数据

#### 场景 2.2：连接失败
- **前提** 网络不通或账号密码错误
- **操作** 系统启动采集任务
- **结果** 记录错误日志，发送告警通知，周期性重试

### 需求 3：SQL Server 性能指标监控

系统应采集以下 SQL Server 性能指标，采集频率默认为 60 秒/次。

| 指标类别 | 具体指标 |
|---------|---------|
| CPU | SQL Server 进程 CPU 使用率、系统总 CPU 使用率 |
| 内存 | SQL Server 内存使用量(MB)、缓冲池大小、缓存命中率 |
| 连接 | 当前用户连接数、最大连接数、活跃会话数 |
| I/O | 磁盘读写延迟、事务吞吐量 |
| 数据库状态 | 数据库文件大小、日志文件大小、自动增长事件 |

#### 场景 3.1：正常采集
- **前提** 数据库连接正常
- **操作** 按采集周期执行查询
- **结果** 指标数据写入 PostgreSQL，Web 面板实时更新

### 需求 4：服务器操作系统内存监控

系统应采集数据库服务器的操作系统内存状态。

| 指标 | 说明 |
|-----|------|
| 总物理内存 | 服务器总物理内存(GB) |
| 已用内存 | 当前已使用的内存量 |
| 可用内存 | 当前可用内存量 |
| 内存使用率 | 已用内存 / 总物理内存 × 100% |
| Page File 使用率 | 页面文件使用情况 |

#### 场景 4.1：内存采集
- **前提** 可以通过 SQL Server 的 `sys.dm_os_performance_counters` 或扩展存储过程获取 OS 信息
- **操作** 执行系统视图查询
- **结果** 内存数据入库并展示

### 需求 5：死锁检测与死锁 SQL 记录

系统应自动检测并记录死锁事件及其详细信息。

| 字段 | 说明 |
|-----|------|
| 发生时间 | 死锁事件发生的精确时间 |
| 死锁 XML | 系统捕获的死锁 XML 报告 |
| 受害会话 | 被选为牺牲品的会话 ID |
| 死锁 SQL | 参与死锁的各条 SQL 语句 |
| 涉及对象 | 死锁涉及的表/索引 |
| 事务信息 | 事务隔离级别、事务开始时间 |

#### 场景 5.1：检测死锁
- **前提** 已启用 SQL Server 死锁捕获（开启 1222/1204 跟踪标志或使用扩展事件）
- **操作** 系统实时监听从目标服务器产生的死锁事件
- **结果** 死锁详情被解析并存储到 PostgreSQL

#### 场景 5.2：死锁告警
- **前提** 检测到新的死锁事件
- **操作** 触发告警
- **结果** 通过 Web 面板突出显示，并可选邮件/钉钉通知

### 需求 6：Web 仪表盘

系统应提供基于 Web 的可视化仪表盘。

#### 场景 6.1：Dashboard 查看
- **前提** 用户访问 Web 界面
- **操作** 展示实时监控面板
- **结果** 用户可以看到 CPU/内存/连接/死锁等图表和统计数据

#### 场景 6.2：死锁历史查询
- **前提** 用户进入死锁分析页面
- **操作** 按时间范围检索死锁记录
- **结果** 展示死锁列表、详情和关联 SQL

### 需求 7：数据存储

采集到的监控数据统一存储在 PostgreSQL 中。

| 数据类型 | 数据表/结构 | 保留期限 |
|---------|------------|---------|
| 性能指标 | 时序表（按指标分类） | 30 天 |
| 死锁事件 | 关系表（死锁主表 + SQL 明细表） | 90 天 |
| 告警日志 | 关系表 | 180 天 |
| 系统配置 | 配置表 | 永久 |

### 需求 8：告警通知

| 告警规则 | 阈值 | 通知方式 |
|---------|------|---------|
| 内存使用率过高 | > 85% 持续 5 分钟 | 页面告警 + 邮件/钉钉 |
| 死锁发生 | 每次检测到死锁 | 页面告警 + 邮件/钉钉 |
| 采集任务中断 | 连续 3 次采集失败 | 页面告警 + 邮件/钉钉 |

### 需求 9：Docker 容器化部署（未来）

未来支持 Docker 部署，需满足：

- 提供 Dockerfile 用于构建监控平台镜像
- 提供 docker-compose.yml 编排监控平台（PG 数据库为外部依赖）
- 环境变量化配置（SQL Server 连接串、PG 连接串、采集间隔等）

## 技术选型建议

| 组件 | 推荐技术 | 说明 |
|-----|---------|------|
| 后端框架 | Python FastAPI | 异步高性能，适合数据采集与 API 服务 |
| SQL Server 驱动 | pymssql / pyodbc | 连接 MS SQL Server |
| PG 驱动 | asyncpg / psycopg2 | 连接 PostgreSQL 后台数据库 |
| 前端框架 | Vue.js + ECharts | 构建监控 Dashboard |
| ORM / SQL | SQLAlchemy（async） | 管理 PostgreSQL 表结构和查询 |
| 任务调度 | APScheduler | 定时采集任务 |
| 数据库迁移 | Alembic | 管理 PG 表结构版本 |
| 告警通知 | smtplib + 钉钉 Webhook | 邮件和即时通讯通知 |
| 容器化 | Docker + docker-compose | 未来部署方案 |

## 安全注意事项

- 数据库连接字符串存储在配置文件/环境变量中，不硬编码在代码中
- 监控平台自身需要认证访问（建议 Basic Auth 或 JWT）
- SQL Server 采集账号（PowerBI）授予最小必要权限：VIEW SERVER STATE、VIEW DATABASE STATE
- PG 账号（u_sv_mgt）授予对应数据库的 CRUD 权限
- 敏感信息（密码）通过环境变量或 Docker Secrets 注入
