# SQL 监控平台 — 指标监控说明文档

## 一、监控架构总览

SQL 监控平台通过 **定时采集 → 数据存储 → 可视化展示 → 告警通知** 的完整链路，实现对 MS SQL Server 关键运行指标的全方位监控。

```
APScheduler（定时调度器，默认 60 秒触发一次）
          │
          ▼
SchedulerManager._collect_and_store()
          │
          ├──→ MetricsCollector（指标采集协调器）
          │        ├── PerformanceCollector  ── 7 大类 / 28 项指标
          │        ├── SlowQueryCollector     ── TOP 20 慢查询
          │        ├── DeadlockDetector       ── 死锁事件
          │        ├── BlockingCollector      ── 阻塞链
          │        ├── DiskSpaceCollector     ── 数据库文件空间
          │        └── IndexAnalyzer          ── 缺失索引 / 索引碎片
          │
          ▼
PostgreSQL 存储（15 张核心表，含指标/慢查询/死锁/告警/配置等）
          │
          ▼
FastAPI 路由层（17 个模块，提供 RESTful JSON 接口）
          │
          ▼
Vue 3 前端展示（仪表盘 + 趋势图 + 告警管理 + 报告导出）
```

### 1.1 调度与采集机制

| 配置项 | 默认值 | 说明 |
|-------|-------|-----|
| 调度驱动 | `APScheduler（AsyncIOScheduler）` | Python 生态最成熟的定时任务框架 |
| 采集间隔 | 60 秒 | 可通过 `SCHEDULER_INTERVAL_SECONDS` 或 系统设置页面调整 |
| 数据清理 | 每天 03:00 | Cron 任务清理过期历史数据 |
| 多实例模式 | 支持 | 开启后遍历 `monitored_instances` 表，为每个活跃实例独立采集 |
| 单实例模式 | 默认 | 使用 `system_configs` 中的 SQL Server 连接配置 |

### 1.2 数据流完整链路

```
SQL Server 目标服务器
      │
      │ pymssql (TDS 协议) 连接
      ▼
MetricsCollector.collect_all()
      │
      ├──→ PerformanceCollector  ── 28 项性能指标
      │     (CPU / 内存 / 连接 / IO / OS内存 / 锁等待 / 批处理请求)
      │
      ├──→ SlowQueryCollector    ── TOP 20 慢查询
      │     (基于 sys.dm_exec_query_stats)
      │
      ├──→ DeadlockDetector      ── 死锁事件
      │     (基于 system_health 扩展事件会话)
      │
      ├──→ BlockingCollector     ── 阻塞链
      │     (基于 sys.dm_exec_requests)
      │
      ├──→ DiskSpaceCollector    ── 数据库文件空间
      │     (基于 sys.master_files)
      │
      └──→ IndexAnalyzer         ── 索引分析
            (缺失索引：sys.dm_db_missing_index_details
             索引碎片：sys.dm_db_index_physical_stats)
      │
      ▼
PostgreSQL 批量写入（asyncpg）
      │
      ▼
AlertEngine.process_metrics() → 告警规则匹配
      │
      ▼
NotificationService.notify_all() → SMTP / 钉钉 / 企业微信
```

## 二、性能指标详解（28 项）

所有指标通过 SQL Server **动态管理视图（DMV）** 和 **性能计数器** 采集。

### 2.1 CPU 指标（2 项）

| 指标名 | 含义 | 单位 | 正常范围 | 数据来源 |
|-------|-----|-----|---------|---------|
| `cpu_usage` | 系统整体 CPU 使用率 | % | 0 ~ 70 | `sys.dm_os_ring_buffers`（RING_BUFFER_SCHEDULER_MONITOR） |
| `sql_cpu` | SQL Server 进程占用的 CPU 使用率 | % | 0 ~ 70 | `sys.dm_os_sys_info` + `sys.dm_os_schedulers` |

**说明与告警建议：**
- 持续 > 85%：可能存在 CPU 压力，需结合慢查询和缺失索引分析
- 持续 > 95%：应立即排查，可能是大量编译/重编译或缺失索引导致

### 2.2 内存指标（4 项）

| 指标名 | 含义 | 单位 | 正常范围 | 数据来源 |
|-------|-----|-----|---------|---------|
| `sql_server_memory_mb` | SQL Server Buffer Pool 已分配的内存 | MB | 根据 `max_server_memory` 配置 | `sys.dm_os_performance_counters` (Total Server Memory) |
| `buffer_cache_hit_ratio` | 缓冲区缓存命中率 | % | ≥ 95（越高越好） | `sys.dm_os_performance_counters` (Buffer cache hit ratio) |
| `target_memory_mb` | SQL Server 目标内存上限 | MB | ≈ max_server_memory 配置 | `sys.dm_os_performance_counters` (Target Server Memory) |
| `page_life_expectancy` | 数据页在内存中保留的预期时间 | 秒 | ≥ 300（越大越好） | `sys.dm_os_performance_counters` (Page life expectancy) |

**说明与告警建议：**
- `buffer_cache_hit_ratio < 90%`：表明内存不足，SQL Server 频繁从磁盘读取
- `page_life_expectancy < 300`：强烈的内存压力信号，应考虑增加服务器内存或优化查询
- `total_server_memory ≈ target_server_memory`：SQL Server 已使用到上限，需关注

### 2.3 连接指标（4 项）

| 指标名 | 含义 | 单位 | 正常范围 | 数据来源 |
|-------|-----|-----|---------|---------|
| `total_connections` | 当前活跃连接总数 | 个 | < 最大连接数 | `sys.dm_exec_connections` COUNT(*) |
| `active_sessions` | 当前正在运行的会话数 | 个 | ≈ 活跃 CPU 核数 | `sys.dm_exec_sessions` WHERE status = 'running' |
| `user_connections` | 用户连接计数器 | 个 | 稳定水平 | `sys.dm_os_performance_counters` (User Connections) |
| `user_processes` | 用户进程数 | 个 | ≈ user_connections | `sys.dm_exec_sessions` WHERE is_user_process = 1 |

**说明与告警建议：**
- 连接数持续增长且未释放：可能存在应用连接池泄漏或长事务未提交
- `active_sessions` 持续 > CPU 核数 × 8：存在明显的阻塞或资源竞争

### 2.4 IO 指标（4 项）

| 指标名 | 含义 | 单位 | 正常范围 | 数据来源 |
|-------|-----|-----|---------|---------|
| `avg_read_latency_ms` | 平均读取延迟 | ms | < 20 | `sys.dm_io_virtual_file_stats` |
| `avg_write_latency_ms` | 平均写入延迟 | ms | < 5 | `sys.dm_io_virtual_file_stats` |
| `total_reads` | 自启动以来累计读取次数 | 次 | — | `sys.dm_io_virtual_file_stats` |
| `total_writes` | 自启动以来累计写入次数 | 次 | — | `sys.dm_io_virtual_file_stats` |

**说明与告警建议：**
- `avg_read_latency_ms > 50`：磁盘读取压力大，考虑升级存储或优化查询减少物理读
- `avg_write_latency_ms > 20`：写入瓶颈，检查事务日志文件大小与自动增长配置

### 2.5 操作系统内存指标（3 项）

| 指标名 | 含义 | 单位 | 正常范围 | 数据来源 |
|-------|-----|-----|---------|---------|
| `total_physical_memory_gb` | 服务器物理内存总量 | GB | 固定值 | `sys.dm_os_sys_info` |
| `available_physical_memory_gb` | 操作系统当前可用内存 | GB | ≥ 10% 总量 | `sys.dm_os_sys_memory` |
| `memory_usage_pct` | 整体内存使用率 | % | < 85 | 计算值 = (total - available) / total × 100 |

**说明与告警建议：**
- `memory_usage_pct > 85%`：触发 `memory_high` 内置告警（critical 级别）
- 可用内存持续不足：可能需要限制 `max_server_memory` 或增加服务器内存

### 2.6 锁等待指标（3 项）

| 指标名 | 含义 | 单位 | 正常范围 | 数据来源 |
|-------|-----|-----|---------|---------|
| `waiting_locks` | 当前处于等待状态的锁数量 | 个 | 0 ~ 少量 | `sys.dm_tran_locks` WHERE request_status = 'WAIT' |
| `lock_waits` | 当前锁等待任务数 | 个 | 0 ~ 少量 | `sys.dm_os_waiting_tasks` WHERE wait_type LIKE 'LCK%' |
| `avg_lock_wait_ms` | 平均锁等待时间 | ms | < 100 | `sys.dm_os_waiting_tasks` AVG(wait_duration_ms) |

**说明与告警建议：**
- 持续出现锁等待：结合阻塞链监控和死锁记录分析原因
- 典型原因：长事务未提交、缺少索引导致扫描范围锁、重复读/可序列化隔离级别

### 2.7 批处理请求指标（3 项）

| 指标名 | 含义 | 单位 | 正常范围 | 数据来源 |
|-------|-----|-----|---------|---------|
| `batch_requests_sec` | 每秒批处理请求数 | count/sec | 稳定水平 | `sys.dm_os_performance_counters` (Batch Requests/sec) |
| `sql_compilations_sec` | 每秒 SQL 编译数 | count/sec | < batch 的 10% | `sys.dm_os_performance_counters` (SQL Compilations/sec) |
| `sql_recompilations_sec` | 每秒 SQL 重编译数 | count/sec | < 编译数的 10% | `sys.dm_os_performance_counters` (SQL Re-Compilations/sec) |

**说明与告警建议：**
- `sql_compilations_sec` 接近 `batch_requests_sec`：存在大量未参数化的查询，浪费 CPU
- `sql_recompilations_sec` 异常高：可能是查询提示、统计信息过期或临时表使用问题

## 三、专项监控模块

### 3.1 慢查询监控（TOP 20）

**采集源：** `sys.dm_exec_query_stats` + `sys.dm_exec_sql_text`

**核心字段：**

| 字段 | 含义 |
|------|-----|
| `sql_hash` | SQL 文本 MD5 哈希值（用于去重聚合） |
| `sql_text` | 完整的 SQL 语句文本 |
| `execution_count` | 自 SQL Server 启动以来累计执行次数 |
| `total_cpu_ms` | 累计消耗 CPU 时间（毫秒） |
| `total_logical_reads` | 累计逻辑读次数 |
| `total_elapsed_ms` | 累计耗时（毫秒） |
| `avg_elapsed_ms` | 单次平均耗时 = total_elapsed_ms / execution_count |
| `last_execution_time` | 最后一次执行时间 |
| `collected_at` | 本次采集时间 |
| `server_address` | 被监控实例标识 |

**工作机制：**
1. 每次采集按 `total_elapsed_ms` 倒序取 TOP 20
2. 对 `sql_text` 计算 MD5 哈希，用于前端聚合展示
3. 每次采集写入新记录（保留历史快照，便于趋势分析）
4. 前端可按 `sql_hash` 聚合查看同一条 SQL 的历史执行情况

**优化建议：**
- 关注 `avg_elapsed_ms` 高（> 1000ms）且执行次数多的 SQL
- 结合"缺失索引"建议检查是否缺少合适索引
- `total_logical_reads` 极高：考虑优化查询逻辑或调整索引策略

### 3.2 死锁监控

**采集源：** `sys.dm_xe_session_targets`（`system_health` 扩展事件会话）

**采集机制：**
1. 从 `system_health` 会话的 `event_file` 目标中读取 `xml_deadlock_report` 事件
2. 解析死锁 XML：
   - 提取 `victim`（被牺牲的会话）
   - 提取 `process-list`（参与死锁的所有会话及 SQL）
   - 提取 `resource-list`（被争用的资源）
3. 对比 `occur_at` 避免重复记录
4. 写入 `deadlocks` 主表 + `deadlock_sqls` 明细表（1:N 关系）

**字段：**
- 主表：`id / occur_at / deadlock_xml / victim_session_id / server_address / analysis_result / created_at`
- 明细表：`id / event_id / session_id / sql_text / isolation_level / involved_objects / created_at`

**AI 分析功能：**
- 前端点击死锁事件详情中的"AI 分析"
- 后端将死锁 XML 和参与会话的 SQL 发送到 DeepSeek LLM
- 获取死锁原因、推荐的索引优化、事务调整建议
- 分析结果写入 `analysis_result` 字段并展示

**前置条件：**
- SQL Server 的 `system_health` 扩展事件会话必须启用（默认已启用）
- 需要配置 DeepSeek API Key（如需 AI 分析）

### 3.3 阻塞链监控

**采集源：** `sys.dm_exec_requests` WHERE `blocking_session_id > 0`

**关键信息：**
- `blocking_session_id`：阻塞源头会话
- `blocked_session_id`：被阻塞的会话
- `blocking_sql_text`：阻塞源执行的 SQL（从 `sys.dm_exec_sql_text` + `sys.dm_exec_connections` 关联获取最末 SQL）
- `blocked_sql_text`：被阻塞会话执行的 SQL
- `wait_type`：等待类型（如 LCK_M_S、LCK_M_X、PAGEIOLATCH_SH 等）
- `wait_time_seconds`：已等待时间（秒）
- `server_address`：实例标识
- `collected_at`：采集时间

**展示与分析：**
- 前端按 `wait_time_seconds` 降序排列，突出最严重的阻塞
- 支持历史记录查询，便于回顾何时发生了持续阻塞
- 阻塞持续 > 60 秒可能触发采集中断相关告警

### 3.4 磁盘空间监控

**采集源：** `sys.master_files` + 数据库维度汇总

**核心字段：**

| 字段 | 含义 |
|------|-----|
| `database_name` | 数据库名称 |
| `file_type` | 文件类型（ROWS 数据文件 / LOG 日志文件） |
| `file_name` | 文件名 |
| `physical_name` | 文件完整物理路径 |
| `size_mb` | 文件当前大小（MB） |
| `used_mb` | 已使用空间（MB） |
| `free_mb` | 可用空间（MB） |
| `usage_pct` | 使用率百分比 |
| `server_address` | 实例标识 |

**采集机制：**
- 读取 `sys.master_files`（所有数据库的所有文件）
- 关联数据库 DB_ID 获取当前文件空间使用
- 按数据库维度汇总 data/log 文件的使用情况

**典型问题：**
- `usage_pct > 90%`：数据库文件空间快用完，可能触发自动增长（默认开启）
- 日志文件持续增长：可能存在长事务或日志备份未定期执行（完整/大容量日志恢复模式）

### 3.5 索引分析

#### 3.5.1 缺失索引（Missing Indexes）

**采集源：** `sys.dm_db_missing_index_details` + `sys.dm_db_missing_index_group_stats` + `sys.dm_db_missing_index_columns`

**核心字段：**

| 字段 | 含义 |
|------|-----|
| `database_name` | 数据库名 |
| `table_name` | 表名 |
| `equality_columns` | 等号查询可使用的列（适合放在索引前部） |
| `inequality_columns` | 不等号查询可使用的列（适合放在等号列之后） |
| `included_columns` | 覆盖索引推荐的包含列 |
| `avg_total_user_cost` | 平均总用户成本（该查询的成本基准） |
| `avg_user_impact` | 预期性能提升百分比（核心评估指标） |
| `user_seeks` | 预计受益的用户查找次数 |
| `user_scans` | 预计受益的用户扫描次数 |

**使用建议：**
- 优先关注 `avg_user_impact` 高（> 80%）且 `user_seeks` 大的建议
- 结合"慢查询"中频繁出现的表进行交叉验证
- 不要盲目跟随建议，避免同一表创建过多索引（影响写入性能）

#### 3.5.2 索引碎片（Index Fragmentation）

**采集源：** `sys.dm_db_index_physical_stats`

**核心字段：**

| 字段 | 含义 |
|------|-----|
| `database_name` | 数据库名 |
| `table_name` | 表名 |
| `index_name` | 索引名 |
| `index_type` | 索引类型（CLUSTERED / NONCLUSTERED / HEAP） |
| `avg_fragmentation_pct` | 平均碎片率（百分比） |
| `page_count` | 索引页数（反映索引大小） |

**筛选条件：**
- `avg_fragmentation_pct > 5`：有一定碎片，关注
- `page_count > 100`：排除过小的索引（碎片意义不大）

**处理建议：**
| 碎片率 | 建议操作 |
|-------|---------|
| 5% ~ 30% | `ALTER INDEX REORGANIZE`（重组，低开销在线操作） |
| > 30% | `ALTER INDEX REBUILD`（重建，更高开销，推荐离线或使用 ONLINE=ON） |

## 四、数据存储与保留策略

### 4.1 数据表清单

| 表名 | 数据量级 | 建议保留策略 | 用途 |
|------|---------|------------|-----|
| `metrics` | 大（28 条/分钟/实例） | 30 ~ 90 天 | 7 大类 28 项性能指标历史 |
| `slow_queries` | 大（20 条/分钟/实例） | 30 天 | 慢查询历史快照 |
| `deadlocks` | 中 | 长期保留（审计分析用） | 死锁事件主表 |
| `deadlock_sqls` | 中 | 长期保留，随主表级联 | 死锁 SQL 明细 |
| `blocking_events` | 大 | 30 天 | 阻塞事件历史 |
| `disk_space_records` | 中 | 90 天 | 磁盘空间使用历史 |
| `missing_indexes` | 中 | 30 天 | 缺失索引建议快照 |
| `index_fragmentation` | 中 | 30 天 | 索引碎片记录 |
| `alert_logs` | 中 | 180 天 | 告警事件日志 |
| `alert_rules` | 小 | 永久 | 告警规则配置 |
| `users` | 小 | 永久 | 用户账号信息 |
| `monitored_instances` | 小 | 永久 | 多实例配置 |
| `audit_logs` | 中 | 365 天 | 审计操作日志 |
| `system_configs` | 小 | 永久 | 运行时系统配置 |
| `report_records` | 中 | 90 天 | 用户生成的报告 |

### 4.2 自动清理机制

平台内置数据清理任务（每天凌晨 03:00 执行）：
- 清理 `metrics` 中超过保留天数的记录
- 清理 `slow_queries` 中超过保留天数的记录
- 清理其他历史数据表中的过期数据

### 4.3 metrics 表结构

```
+--------------------+--------------------------+
| 字段                | 说明                     |
+--------------------+--------------------------+
| id                 | 主键自增                  |
| category           | 指标分类（cpu/memory/...） |
| metric_name        | 指标名称（如 cpu_usage）   |
| metric_value       | 指标数值（DOUBLE）         |
| unit               | 单位（%/MB/GB/ms/count）   |
| collected_at       | 采集时间（TIMESTAMPTZ）     |
| server_address     | 实例标识（多实例场景用）     |
| created_at         | 记录创建时间                |
+--------------------+--------------------------+
```

**常见查询模式：**
- 仪表盘实时数据：`WHERE collected_at = (最新时间)`
- 趋势图：`WHERE metric_name = ? AND collected_at BETWEEN ? AND ? ORDER BY collected_at`
- 摘要统计：`MAX / AVG / MIN` 按时间窗口聚合

## 五、告警规则体系

### 5.1 内置告警规则（硬编码，默认生效）

| 规则名称 | 触发条件 | 严重级别 | 冷却期 | 说明 |
|---------|---------|---------|-------|-----|
| `memory_high` | OS 内存使用率 > 85% 且持续 ≥ 5 分钟 | critical | 10 分钟 | 服务器整体内存压力告警 |
| `deadlock_detected` | 本次采集发现新死锁事件 | high | 5 分钟 | 死锁发生后立即告警 |
| `collection_interrupted` | 连续 3 个采集周期无新指标写入 | high | 15 分钟 | 采集任务或 SQL Server 连接异常 |

### 5.2 自定义告警规则

**通过前端告警规则页面配置，可设置：**

| 配置项 | 说明 | 示例 |
|-------|-----|------|
| 规则名称 | 人类可读的标识 | `高 CPU 使用率` |
| 指标分类 | 从 7 大类中选择 | `cpu` |
| 指标名称 | 具体指标名 | `cpu_usage` |
| 比较操作符 | `>` / `<` / `>=` / `<=` / `=` | `>` |
| 阈值 | 触发条件数值 | `90` |
| 严重级别 | `low / medium / high / critical` | `high` |
| 静默时段 | 指定时间段内不触发 | `00:00 - 06:00` |
| 启用状态 | 开关控制规则是否生效 | 启用 |

### 5.3 告警生命周期

```
1. 采集数据写入 PostgreSQL
2. AlertEngine 轮询检查所有规则
   ├─ 对内置规则：检查内存/死锁/采集状态
   └─ 对自定义规则：按 category + metric_name 匹配阈值
3. 若触发：
   ├─ 检查冷却期（避免短时间重复告警）
   ├─ 检查静默时段
   └─ 写入 alert_logs
4. 触发通知服务：
   ├─ SMTP 邮件 → 收件人（ALERT_EMAILS 配置）
   ├─ 钉钉 Webhook → 机器人配置
   └─ 企业微信 Webhook → 机器人配置
5. 用户确认告警 → PUT /api/alerts/{id}/acknowledge
6. 告警状态变为已确认
```

## 六、多实例模式

### 6.1 启用方式

1. 登录前端，进入"系统设置"

### 6.2 实例配置项

| 字段 | 说明 |
|------|-----|
| `name` | 实例名称（前端展示用） |


## 八、采集对源数据库的影响

| 影响项 | 实际情况 | 优化建议 |
|-------|---------|---------|
| CPU | 很低，所有查询均为 DMV 轻量读取 | 在大型实例上提高采集间隔到 120~300 秒 |
| 内存 | 很低 | — |
| 锁 | **不加锁**（DMV 读取不持有用户表锁） | — |
| 连接 | 每次采集建立/复用 1 个短连接 | 启用连接复用 |
| 网络 | 很低（KB 级别数据） | — |

**总体结论：** 采集对源 SQL Server 的性能影响可忽略不计，默认 60 秒的采集间隔适用于绝大多数场景。

## 九、诊断与问题排查

| 问题现象 | 可能原因 | 排查方式 |
|---------|---------|---------|
| 指标卡片全为 0 或空 | 采集任务未执行；SQL Server 连接失败 | 查看 `docker-compose logs backend`；检查 `MSSQL_HOST/PORT/USER/PASSWORD` 配置 |
| 死锁无记录 | `system_health` 会话未启用；SQL Server 版本过低 | 在 SQL Server 执行：`SELECT * FROM sys.dm_xe_sessions WHERE name = 'system_health'` |
| 告警未通知 | 通知渠道未配置；处于冷却期；告警被静默时段屏蔽 | 检查 `DINGTALK_WEBHOOK_URL` / `SMTP_SERVER` 等配置；查看 `alert_logs` 中的 `notification_sent` 字段 |
| 图表加载慢 | 历史数据过多；缺少索引 | 对 `metrics(collected_at)` / `metric_name` 等字段建立索引；执行定期数据清理 |
| 内存指标数值异常 | SQL Server `max_server_memory` 未配置；与操作系统其他应用竞争内存 | 在 SQL Server 执行 `sp_configure 'max server memory'`；确认是否限制了 SQL Server 内存上限 |
| 多实例采集失败 | 实例连接信息错误；SQL Server 防火墙未放行 | 使用"实例管理"页面的"测试连接"按钮逐个验证 |

## 十、附录

### 10.1 SQL Server 监控账号最小权限

```sql
-- 创建登录账号
USE master;
CREATE LOGIN sql_monitor WITH PASSWORD = 'your_strong_password';

-- 创建用户
CREATE USER sql_monitor FOR LOGIN sql_monitor;

-- 授予 VIEW SERVER STATE（查看服务器级 DMV）
GRANT VIEW SERVER STATE TO sql_monitor;

-- 授予 VIEW DATABASE STATE（查看数据库级 DMV，可选，授予每个目标数据库）
-- 对每个需要监控的数据库执行：
-- USE [target_db];
-- CREATE USER sql_monitor FOR LOGIN sql_monitor;
-- GRANT VIEW DATABASE STATE TO sql_monitor;

-- 如需查看存储过程/函数文本（可选，用于慢查询分析）
-- GRANT VIEW ANY DEFINITION TO sql_monitor;
```

### 10.2 关键指标速查卡

| 指标 | 正常范围 | 需关注 | 严重告警 |
|-----|---------|-------|---------|
| CPU 使用率 | < 70% | 70%~90% | > 90% |
| Buffer Cache Hit Ratio | ≥ 95% | 85%~95% | < 85% |
| Page Life Expectancy | ≥ 300 秒 | 100~300 秒 | < 100 秒 |
| 平均读取延迟 | < 20ms | 20~100ms | > 100ms |
| 平均写入延迟 | < 5ms | 5~20ms | > 20ms |
| OS 内存使用率 | < 85% | 85%~90% | > 90% |
| 锁等待数 | < 5 | 5~50 | > 50 |
| SQL 重编译率 | < 编译数的 10% | 10%~30% | > 30% |
