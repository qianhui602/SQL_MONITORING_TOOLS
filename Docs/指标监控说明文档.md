# SQL 监控平台 — 指标监控说明文档

## 一、监控架构总览

```
APScheduler (定时调度器, 默认 60 秒)
    │
    ▼
SchedulerManager._collect_and_store()
    │
    ├──→ MetricsCollector (性能协调器)
    │       ├── PerformanceCollector  ──  7 大类 / 28 项指标
    │       ├── SlowQueryCollector    ──  TOP 20 慢查询
    │       └── DeadlockDetector      ──  死锁事件
    │
    ▼
PostgreSQL 存储 (10 张监控表)
    │
    ▼
FastAPI 路由层 (6 个模块)
    │
    ▼
Vue.js 前端展示
```

## 二、采集机制

### 2.1 调度方式

- **驱动**：APScheduler (AsyncIOScheduler)
- **默认间隔**：每 60 秒执行一次
- **可配置**：通过系统设置页面修改 `采集间隔(秒)` 参数
- **运行模式**：
  - **单实例模式**：使用默认 SQL Server 连接配置
  - **多实例模式**：遍历 `monitored_instances` 表，为每个活跃实例独立采集

### 2.2 采集流程

```
_collect_and_store()
    │
    ├─ 1. 从 system_configs 加载运行时配置
    ├─ 2. 判断单实例/多实例模式
    ├─ 3. 建立 SQL Server 连接 (pymssql)
    ├─ 4. 执行 7 类 DMV 查询采集性能指标
    ├─ 5. 执行慢查询采集 (TOP 20)
    ├─ 6. 执行死锁检测 (XML 解析)
    ├─ 7. 批量写入 PostgreSQL
    └─ 8. 触发告警引擎检查
```

### 2.3 数据隔离

- 多实例模式下每个实例使用独立连接
- 数据通过 `server_address` 字段区分来源（格式：`实例名(host:port)`）

## 三、性能指标详解

所有指标通过 SQL Server 动态管理视图（DMV）采集，共 **7 大类 / 28 项指标**。

### 3.1 CPU 指标

| 指标名称 | 含义 | 单位 | 查询视图 |
|---------|------|------|---------|
| `cpu_usage` | 系统 CPU 使用率 | % | `sys.dm_os_ring_buffers` |
| `sql_cpu` | SQL Server 进程 CPU 占用率 | % | `sys.dm_os_sys_info` |

**采集 SQL**：从 `RING_BUFFER_SCHEDULER_MONITOR` 中提取 `ProcessUtilization`。

### 3.2 内存指标

| 指标名称 | 含义 | 单位 | 查询视图 |
|---------|------|------|---------|
| `sql_server_memory_mb` | SQL Server 已分配内存 | MB | `sys.dm_os_performance_counters` — Total Server Memory (KB) |
| `buffer_cache_hit_ratio` | 缓冲区缓存命中率 | % | `sys.dm_os_performance_counters` — Buffer cache hit ratio |
| `target_memory_mb` | SQL Server 目标内存 | MB | `sys.dm_os_performance_counters` — Target Server Memory (KB) |
| `page_life_expectancy` | 页预期生存时间 | 秒 | `sys.dm_os_performance_counters` — Page life expectancy |

> **说明**：`buffer_cache_hit_ratio` 反映数据缓存效率，通常应 > 95%。`page_life_expectancy` 低于 300 秒表明内存压力较大。

### 3.3 连接指标

| 指标名称 | 含义 | 单位 | 查询视图 |
|---------|------|------|---------|
| `total_connections` | 总连接数 | 个 | `sys.dm_exec_connections` |
| `active_sessions` | 活跃会话数 | 个 | `sys.dm_exec_sessions` WHERE status = 'running' |
| `user_connections` | 用户连接数（计数器） | 个 | `sys.dm_os_performance_counters` — User Connections |
| `user_processes` | 用户进程数 | 个 | `sys.dm_exec_sessions` WHERE is_user_process = 1 |

### 3.4 I/O 指标

| 指标名称 | 含义 | 单位 | 查询视图 |
|---------|------|------|---------|
| `avg_read_latency_ms` | 平均读取延迟 | ms | `sys.dm_io_virtual_file_stats` |
| `avg_write_latency_ms` | 平均写入延迟 | ms | `sys.dm_io_virtual_file_stats` |
| `total_reads` | 总读取次数 | 次 | `sys.dm_io_virtual_file_stats` |
| `total_writes` | 总写入次数 | 次 | `sys.dm_io_virtual_file_stats` |
| `read_mb` | 读取数据总量 | MB | `sys.dm_io_virtual_file_stats` |
| `write_mb` | 写入数据总量 | MB | `sys.dm_io_virtual_file_stats` |

> **说明**：该指标为自 SQL Server 启动以来的累积值，前后对比可判断存储性能趋势。

### 3.5 操作系统内存指标

| 指标名称 | 含义 | 单位 | 查询视图 |
|---------|------|------|---------|
| `total_physical_memory_gb` | 物理内存总量 | GB | `sys.dm_os_sys_info` |
| `available_physical_memory_gb` | 可用物理内存 | GB | `sys.dm_os_sys_memory` |
| `memory_usage_pct` | 内存使用率 | % | 计算值：(total - available) / total |

### 3.6 锁等待指标

| 指标名称 | 含义 | 单位 | 查询视图 |
|---------|------|------|---------|
| `waiting_locks` | 等待中的锁数 | 个 | `sys.dm_tran_locks` WHERE request_status = 'WAIT' |
| `lock_waits` | 锁等待任务数 | 个 | `sys.dm_os_waiting_tasks` WHERE wait_type LIKE 'LCK%' |
| `avg_lock_wait_ms` | 平均锁等待时间 | ms | `sys.dm_os_waiting_tasks` AVG(wait_duration_ms) |

### 3.7 批处理请求指标

| 指标名称 | 含义 | 单位 | 查询视图 |
|---------|------|------|---------|
| `batch_requests_sec` | 每秒批处理请求数 | count/s | `sys.dm_os_performance_counters` — Batch Requests/sec |
| `sql_compilations_sec` | 每秒 SQL 编译数 | count/s | `sys.dm_os_performance_counters` — SQL Compilations/sec |
| `sql_recompilations_sec` | 每秒 SQL 重编译数 | count/s | `sys.dm_os_performance_counters` — SQL Re-Compilations/sec |

> **说明**：`sql_recompilations_sec` 占比过高可能意味着缺少参数化或计划缓存问题。

## 四、专项监控

### 4.1 慢查询监控

| 项目 | 说明 |
|------|------|
| **采集方式** | 每次采集 TOP 20 条总耗时最长的查询 |
| **核心SQL** | `sys.dm_exec_query_stats` + `sys.dm_exec_sql_text` |
| **关键指标** | `sql_text`, `execution_count`, `total_cpu_ms`, `total_elapsed_ms`, `avg_elapsed_ms`, `total_logical_reads` |
| **去重机制** | 对 SQL 文本计算 MD5 哈希，用于前端聚合统计 |
| **保存策略** | 每次采集写入新记录，保留全部历史数据 |

### 4.2 死锁监控

| 项目 | 说明 |
|------|------|
| **采集方式** | 从系统健康会话扩展事件中解析死锁报告 |
| **核心SQL** | `sys.dm_xe_session_targets` WHERE name = 'system_health' |
| **XML解析** | 提取 victim_session_id、参与进程、SQL语句、隔离级别、涉及对象 |
| **AI分析** | 支持一键调用 DeepSeek API 分析死锁原因并给出优化建议 |
| **存储结构** | 主表 `deadlocks` + 明细表 `deadlock_sqls`（一对多） |

### 4.3 阻塞链监控

| 项目 | 说明 |
|------|------|
| **采集方式** | 实时查询当前阻塞链 |
| **核心SQL** | `sys.dm_exec_requests` WHERE blocking_session_id > 0 |
| **关键信息** | block_spid（被阻塞会话）、blocking_spid（阻塞源）、wait_type、wait_time_ms、blocked_sql、blocking_sql |
| **展示** | 前端按 `wait_time_ms` 降序排列，突出严重阻塞 |

### 4.4 磁盘空间监控

| 项目 | 说明 |
|------|------|
| **采集方式** | 查询 master_files 按数据库汇总 |
| **核心SQL** | `sys.master_files` 按 database_id 分组（排除系统库） |
| **关键指标** | `data_file_mb`, `log_file_mb`, `total_mb`, `used_mb`, `free_mb`, `usage_pct` |
| **计算逻辑** | free_mb = total_mb - used_mb; usage_pct = used_mb / total_mb * 100 |

### 4.5 索引分析

#### 缺失索引

| 项目 | 说明 |
|------|------|
| **采集方式** | 查询 SQL Server 自动建议的缺失索引 |
| **核心SQL** | `sys.dm_db_missing_index_details` + `sys.dm_db_missing_index_group_stats` |
| **关键指标** | `avg_user_impact`（预期提升百分比）、`user_seeks/scans` |
| **排序** | 按 `avg_user_impact` 降序排列 |

#### 索引碎片

| 项目 | 说明 |
|------|------|
| **采集方式** | 调用 `sys.dm_db_index_physical_stats` 函数 |
| **过滤条件** | avg_fragmentation_pct > 5% 且 page_count > 100 |
| **索引类型** | 区分 CLUSTERED / NONCLUSTERED |

## 五、数据存储

### 5.1 监控数据表清单

| 表名 | 用途 | 数据量级 | 保留策略 |
|------|------|---------|---------|
| `metrics` | 通用性能指标（7大类28项） | 大 | 建议定期清理 |
| `slow_queries` | 慢查询记录 | 大 | 建议定期清理 |
| `deadlocks` | 死锁事件 | 中 | 建议保留 |
| `deadlock_sqls` | 死锁关联SQL明细 | 中 | 随主表级联 |
| `blocking_events` | 阻塞事件历史 | 大 | 建议定期清理 |
| `disk_space_records` | 磁盘空间历史 | 中 | 建议定期清理 |
| `missing_indexes` | 缺失索引建议 | 中 | 建议定期清理 |
| `index_fragmentation` | 索引碎片记录 | 中 | 建议定期清理 |

### 5.2 `metrics` 表结构

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER (PK) | 主键 |
| category | VARCHAR(50) | 分类：cpu / memory / connection / io / os_memory / locks / batch_requests |
| metric_name | VARCHAR(100) | 指标名称，如 cpu_usage |
| metric_value | DOUBLE | 指标数值 |
| unit | VARCHAR(20) | 单位：%、MB、GB、ms、count |
| collected_at | TIMESTAMPTZ | 采集时间 |
| server_address | VARCHAR(255) | 实例标识 |

## 六、告警规则

### 6.1 内置告警

| 告警名称 | 触发条件 | 严重级别 | 冷却期 |
|---------|---------|---------|-------|
| 内存过高 | OS 内存使用率 > 85% 且持续 ≥ 5 分钟 | critical | 10 分钟 |
| 检测到死锁 | 死锁事件表中出现新记录 | high | 5 分钟 |
| 采集中断 | 连续 3 个采集周期无新数据 | high | 15 分钟 |

### 6.2 自定义告警

通过告警规则页面可创建自定义告警，支持：
- **指标选择**：按分类 + 指标名匹配任意采集指标
- **条件运算符**：`>` / `<` / `>=` / `<=` / `=`
- **静默时段**：支持配置指定时间段不触发告警
- **多渠道通知**：系统通知 + 企业微信机器人

## 八、性能与安全

### 8.1 采集对源库的影响

- 所有查询均针对 **DMV（动态管理视图）** 和 **系统目录视图**，不锁定用户表
- 通过 `pymssql` 建立独立连接，采集完成后立即关闭
- 默认 60 秒的采集间隔不会对源库造成显著负载

### 8.2 连接安全

- SQL Server 连接信息存储在 PostgreSQL 中，数据库中密码无加密
- 建议为监控账号授予最小权限：`VIEW SERVER STATE` + `VIEW DATABASE STATE` 即可

### 8.3 数据存储

- 监控数据存储在独立的 PostgreSQL 数据库中
- 建议对 `metrics`、`slow_queries` 等大表按时间分区并定期清理过期数据
